# Домен: Products Domain

## Опис домену

Даний домен відповідає за збереження інформації про товари магазину та їх прив'язку до інших таблиць.

## Основні бізнес-правила домену

* Перше головне правило > товар належить одній категорії або не належить жодній.
* Друге головне правило > варіант товару завжди належить лише одному товару.
* Третє головне правило > один товар може мати багато варіантів.
* Четвертве головне правило > значення опцій можуть використовуватися у багатьох варіантах товарів.
* П'яте головне правило > `slug` категорії, бренду та товару має бути унікальним.
* Шосте головне правило > користувач може залишити лише один відгук про товар.
* Сьоме головне правило > товар не може існувати без варіанта та навпаки.

## Архітектурний контекст

### Типові сценарії використання

1. Перегляд каталогу товарів.
2. Вибір конкретного варіанта товару.

## Таблиця: categories

Зберігає інформацію про категорії товарів магазину.

### Основні поля

| Поле         | Тип               | NULL  | Foreign Key (FK) | Опис                                             |
|:-------------|:------------------|:------|:-----------------|:-------------------------------------------------|
| `id`         | `bigint unsigned` | `NO`  |                  | Унікальний ідентифікатор категорії.              |
| `name`       | `varchar(255)`    | `NO`  |                  | Назва категорії.                                 |
| `slug`       | `varchar(255)`    | `NO`  |                  | Унікальний slug категорії.                       |
| `parent_id`  | `bigint unsigned` | `YES` | `categories(id)` | Унікальний ідентифікатор батьківської категорії. |
| `created_at` | `timestamp`       | `YES` |                  | Дата створення запису.                           |
| `updated_at` | `timestamp`       | `YES` |                  | Дата оновлення запису.                           |

### Зв'язки (Relationships)

* `One To Many` > `products` Одна категорія може мати багато товарів.
* `Has Many Through` > `product_variants` Одна категорія може мати багато варіантів товарів через таблицю `products`.

### Індекси (Indexes)

* `UNIQUE(slug)` *Причина:* `slug` категорії це унікальна частина (ідентифікатор) в URL для зручного читання людиною.

### Обмеження цілісності

* **Foreign Key: `parent_id` REFERENCES `categories(id)` ON DELETE SET NULL (якщо видалити батьківську категорію, поле `parent_id` стає `null`, але дочірня категорія не видаляється).**

### Особливості продуктивності

* *Потенційні проблеми:* На початковому етапі таблиця не має критичного навантаження.
* *Можливі оптимізації:* Індексація поля `parent_id` (ще не реалізовано).

### Де реалізовано

- `app/Models/Product/Category`.

## Таблиця: products

Зберігає базову інформацію про товари для продажу.

### Основні поля

| Поле          | Тип               | NULL  | Foreign Key (FK) | Default | Опис                               |
|:--------------|:------------------|:------|:-----------------|:--------|:-----------------------------------|
| `id`          | `bigint unsigned` | `NO`  |                  |         | Унікальний ідентифікатор товару.   |
| `category_id` | `bigint unsigned` | `YES` | `categories(id)` |         | Унікальний ідентифікатор категоії. |
| `brand_id`    | `bigint unsigned` | `YES` | `brands(id)`     |         | Унікальний ідентифікатор бренду.   |
| `name`        | `varchar(255)`    | `NO`  |                  |         | Назва товару.                      |
| `slug`        | `varchar(255)`    | `NO`  |                  |         | Унікальний slug товару.            |
| `description` | `varchar(255)`    | `YES` |                  |         | Опис товару.                       |
| `image`       | `varchar(255)`    | `YES` |                  |         | Зображення товару.                 |
| `is_active`   | `tinyint(1)`      | `NO`  |                  | `1`     | Статус активності товару.          |
| `created_at`  | `timestamp`       | `YES` |                  |         | Дата створення запису.             |
| `updated_at`  | `timestamp`       | `YES` |                  |         | Дата оновлення запису.             |
| `deleted_at`  | `timestamp`       | `YES` |                  |         | Дата видалення запису.             |

### Зв'язки (Relationships)

* `Many To One` > `categories` Один товар може належати тільки до одної категорії.
* `Many To One` > `brands` Один товар може належати тільки до одного бренду.
* `One To Many` > `product_variants` Один товар має щонайменше один варіант (або багато).
* `Many To Many` > `options` Один товар може належати до багатьох опцій.
* `One To Many` > `reviews` Один товар може мати багато відгуків.

### Індекси (Indexes)

* `UNIQUE(slug)` *Причина:* `slug` це унікальна частина (ідентифікатор) в URL для зручного читання людиною.

### Обмеження цілісності

* **Foreign Key: `category_id` REFERENCES `categories(id)` ON DELETE SET NULL (якщо видалити категорію, поле `category_id` стає `null`, але товар не видаляється).**
* **Foreign Key: `brand_id` REFERENCES `brands(id)` ON DELETE SET NULL (якщо видалити бренд, поле `brand_id` стає `null`, але товар не видаляється).**

### Особливості продуктивності

* *Потенційні проблеми:* На початковому етапі таблиця не має критичного навантаження.
* *Можливі оптимізації:* Індексація поля `category_id`, `brand_id` та `is_active` (ще не реалізовано).

### Де реалізовано

- `app/Models/Product/Product`.

## Таблиця: product_options

Проміжна таблиця для зв'язку товарів та опцій (які опції доступні для конкретного товару).

### Основні поля

| Поле         | Тип               | NULL  | Foreign Key (FK) | Опис                                 |
|:-------------|:------------------|:------|:-----------------|:-------------------------------------|
| `id`         | `bigint unsigned` | `NO`  |                  | Унікальний ідентифікатор комбінації. |
| `product_id` | `bigint unsigned` | `NO`  | `products(id)`   | Унікальний ідентифікатор товару.     |
| `option_id`  | `bigint unsigned` | `NO`  | `options(id)`    | Унікальний ідентифікатор опції.      |

### Індекси (Indexes)

* `UNIQUE(product_id, option_id)` *Причина:* забезпечення того, що один товар не може мати 2 однакових опції.

### Обмеження цілісності

* **Foreign Key: `product_id` REFERENCES `products(id)` ON DELETE CASCADE (якщо видалити товар, його запис видаляється).**
* **Foreign Key: `option_id` REFERENCES `options(id)` ON DELETE CASCADE (якщо видалити опцію, її запис видаляється).**

## Таблиця: reviews

Зберігає інформацію про відгуки користувачів про товари.

### Основні поля

| Поле         | Тип                | NULL  | Foreign Key (FK) | Опис                                  |
|:-------------|:-------------------|:------|:-----------------|:--------------------------------------|
| `id`         | `bigint unsigned`  | `NO`  |                  | Унікальний ідентифікатор відгуку.     |
| `user_id`    | `bigint unsigned`  | `NO`  | `users(id)`      | Унікальний ідентифікатор користувача. |
| `product_id` | `bigint unsigned`  | `NO`  | `products(id)`   | Унікальний ідентифікатор товару.      |
| `rating`     | `tinyint unsigned` | `NO`  |                  | Оцінка користувача про товар.         |
| `comment`    | `text`             | `YES` |                  | Коментар користувача про товар.       |
| `created_at` | `timestamp`        | `YES` |                  | Дата створення запису.                |
| `updated_at` | `timestamp`        | `YES` |                  | Дата оновлення запису.                |

### Зв'язки (Relationships)

* `Many To One` > `users` Один відгук може належати лише одному користувачу.
* `Many To One` > `products` Один відгук може належати лише одному товару.

### Індекси (Indexes)

* `UNIQUE(user_id, product_id)` *Причина:* забезпечення того, що користувач не може залишити більше одного відгуку про товар.

### Обмеження цілісності

* **Foreign Key: `user_id` REFERENCES `users(id)` ON DELETE CASCADE (якщо видалити користувача, його відгук про товар видаляється).**
* **Foreign Key: `product_id` REFERENCES `products(id)` ON DELETE CASCADE (якщо видалити товар, відгуки про нього видаляються).**

### Особливості продуктивності

* *Потенційні проблеми:* На початковому етапі таблиця не має критичного навантаження.
* *Можливі оптимізації:* Індексація поля `rating` (ще не реалізовано).
